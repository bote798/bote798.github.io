坑 1：rollout history 只显示两条 → 以为”只能存两个版本”

现象

来回更新镜像 + 回滚了好几次，rollout history 却始终只有两条，而且 REVISION 号是 4、5 这种大数字：

第一反应：”Deployment 只能保留两个版本？”

真相

history 条数 = 当前保留的 ReplicaSet 个数，而 RS 是按”Pod 模板（主要是镜像）”去重的，不是按操作次数记流水账。

只用过两种镜像，就只会存在两个 RS，history 自然最多两条：

来回 undo 并不新建 RS，只是让目标 RS 的副本数升回来、revision 号往上跳。

为什么号是 4、5 不是 1、2、3

回滚会作废旧号、发一个新号。号只增不减，但底层还是那两个 RS 轮流坐庄：

同一个模板，谁最近被用就顶当前最大号，旧号被作废。

坑 2：集群删掉后 kubectl 报 connection refused (localhost:8080)

现象

原因

kubectl 没读到任何有效的 kubeconfig 上下文时，会**回退到默认地址 localhost:8080**。集群被删（或虚拟机重启后 Kind 容器没起、kubeconfig 丢失）后就是这个现象——不是端口配错，是压根没连上任何集群。

判断与解决

检查 命令 说明
Kind 还有没有集群 kind get clusters 空 = 集群没了，需重建
有没有残留容器 docker ps -a | grep kind 无输出 = 删干净了
有没有上下文 kubectl config get-contexts 没有 kind-xxx = kubeconfig 丢了
  • 集群还在、只是上下文丢了：kind export kubeconfig --name k8s-lab
  • 集群整个没了：用带镜像加速的配置重建（见第一章坑 3）

坑 3：kind get clusters 敲成 kubectl get clusters

clusters 不是 kubectl 的资源类型，kubectl 会试图连 API server 查，于是报连接错误，跟集群状态无关。

记牢分工：

命令前缀 管什么
kind ... 管集群本身（建、删、列集群、导出 kubeconfig）
kubectl ... 管集群里的资源（Pod、Deployment、Service 等）

扩展知识：revisionHistoryLimit —— 真正的”留几个”开关

history 能留几条，由 Deployment 的 revisionHistoryLimit 决定（默认 10）：

取值 效果
10（默认） 最多保留 10 个旧 RS，超出从最老的清
3~5 生产常用，够回滚又不囤太多空 RS
0 不留任何旧 RS，无法回滚

注意：这个上限限制的是”不同模板的旧 RS 个数”，和坑 1 的去重机制叠加——你只用两种镜像时，无论上限设多大，也只有两个 RS。

扩展知识：CHANGE-CAUSE 为什么总是
