为什么不能把配置写进镜像？

回顾前三章你造的 Nginx：

这个 nginx 用的是什么首页？镜像里自带的默认页（”Welcome to nginx!”）。端口？镜像里写死的 80。

需求 写进镜像能做到吗？
换个首页文案（不改代码） ✗ 要重新 docker build
开发连 dev 数据库，生产连 prod 数据库 ✗ 要打两个镜像
改个端口号 ✗ 要改 Dockerfile 重新构建

核心矛盾：镜像 = 不变的（二进制+库），配置 = 会变的。K8s 的解法：ConfigMap。

ConfigMap 是什么

ConfigMap 就是一个存在集群里的 key-value 字典。Pod 引用它，但不拥有它。删 Pod → ConfigMap 还在。

创建 ConfigMap 的三种方式

方式一：–from-literal（从命令行字面量）

literal = “字面量”，值直接写在命令行里。等号左边是 KEY（你起名），右边是 VALUE（随便写）。

方式二：–from-file（从文件）

文件名变成 key（index.html），文件内容变成 value。适合 HTML、nginx.conf 等长内容。

坑 2：以为更新 ConfigMap 后环境变量也会变

现象

用 envFrom 或 valueFrom 注入的 Pod，更新 ConfigMap 后变量值不变。

真相

ConfigMap 更新后：

注入方式 是否自动更新
volume mount（文件挂载） ✅ 会自动更新（kubelet 定期同步，默认 ~1 分钟）
envFrom / valueFrom（环境变量） ❌ 不会更新（Pod 启动时只注入一次）

验证

环境变量只在 Pod 启动时注入一次，之后 ConfigMap 怎么改都不会变。需要新值 → 重建 Pod。

扩展知识：三种注入方式的选择指南

场景 推荐方式 理由
数据库地址、端口、环境名 envFrom 一键全量注入，省事
某几个敏感/关键变量需重命名 valueFrom 变量名可以和 key 不同
配置文件（nginx.conf、html） volume mount 文件形式，支持热更新
只替换目录里一个文件 volume mount + subPath 不覆盖目录其他文件

练习 2：valueFrom 挑选注入

练习 3：文件挂载 + 覆盖验证
